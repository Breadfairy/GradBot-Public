# GradBot – CLI Crypto Backtest Environment..

## Preamble
This project is an exploration into the limits of agentic coding, specifically,
Codex. With the ongoing massive layoffs in software development due to rising
LLM assistance, it seemed fitting to become as familiar as possible with the
technology.

So, as a disclaimer, the majority of the codebase has been generated using
Codex. However, the overall architecture, design decisions and logic have been
carefully curated by myself and remain my intellectual property.

Consider everything outside of this readme to be generated by Codex.

-----------------------------------------------------------------------------

## Goal
- develop a backtest environment and trading algorithm that can capture the
  macro swing trends of the LINK/USDT pair on Binance. Focusing specifically 
  on gradients and similar mathematical signals derived from moving averages. 
  At the same time, explore the limits of agentic coding.

## Objectives
- develop backtesting environment
- develop trading configuration and flagging logic
- deploy paper trading bot on Binance Testnet
- deploy live trading bot on Binance Mainnet
- migrate local algorithm to cloud environment for better performance and
  uptime.

## Status
- Current state: exploring flagging of rollover and ramp-up signals in an
  attempt to capture the beginnings of macro trend reversals. Successful
  backtesting of strategy and exploring refinements.
- current "highscore" renders ~100% increase over benchmark, performing well in
  the sideways 'crab' macro trend and downtrend reversal; however
  underperforming in capturing uptrend reversals due to partial asset exposure.

## Demo
Requirements
- Python 3.10+
- Packages: `python-binance`, `numpy`, `pandas`, `matplotlib`
- Binance API keys are needed for kline cache update scripts: `src/config.ini`
  (copy `src/config.example.ini`).

Warning: The program will not run without a Binance API key. The API is used
to pull the historical kline data from the exchange. Without it, there will
be no data to iterate through. If you don't have a personal Binance API, see
`outputs/tuning/example/` for pre-generated results.

1. Use `src/config.example.ini` and enter your own key and secret. Rename it to
   `config.ini` and place it in `src/`.
2. Navigate to scripts/ and run: `./run_tune.sh template-config.json "demo"`
   This will execute the tuning sweep process which finds a "best" config.

### Example:

```text
              ./run_tune.sh template-config.json "Example"
              [warning] Cache size: 995.08 MB
              [##################################----------------]  68.23% (786/1152) ETA 00:00:28
```
Here, we have `./run_tune.sh <profile.json> <out_dir>`. The template config
partially shown below has 1152 unique configurations built from the Cartesian
product of each key's array of test values. Each of these configs gets
backtested on the predefined tuner_days period. In this case, 730 days of 15m
and 6h candles.

```text
           {
             "_comment": "LINKUSDT-config",
             "tickers": ["LINKUSDT"],
             "primer_days": 180,
             "tuner_days": 730,
             "holdout_days": 365,
             "intervals": ["15m"],
             "p1": 12,
             "p2": 20,
             "p3": 55,
             "GRAD1_BUY_Z_MIN": [1.1, 1.4, 1.8],
             "GRAD1_SELL_Z_MIN": [1.2, 1.5, 1.7],
             "GRAD1_BUY_WIN_DAYS": 365,
             "GRAD1_SELL_WIN_DAYS": 365,
             "SPACING_Z_MIN_12": [0.4, 0.8, 1.0],
             "SPACING_Z_MIN_23": [0.5, 0.7, 0.8],
             "SPACING_WIN_DAYS_12": [365],
             "SPACING_WIN_DAYS_23": [365],
             ...
             "MACRO_INTERVAL": "6h",
             ...
             }
```

The algorithm selects the specific config associated with the highest %
increase in gross value vs benchmark, generates some visual aids and prints the
associated statistics of that profile. Another profile "stats" is also printed, 
which is a risk minimised profile selected from a weighted scoring of the risk metrics.

```text

              [tune] preparing charts...
              =======================================================
              [tune] GrossVhodl: best
key metric! > [tune] - LINKUSDT: +42.27%  TRADES: 61
              [tune] - EDGE%: +38.14%, HODL%:  -2.90%
              [tune] - BUYS: 32, SELLS: 29
              [tune] RiskStats
              [tune] - SHARPE 1w  (abs/rel): 0.99 / 0.83
              [tune] - SHARPE 4w  (abs/rel): 1.03 / 1.21
              [tune] - SHARPE 13w (abs/rel): 0.60 / 0.65
              [tune] - SORTINO 1w  (abs/rel): 1.51 / 1.25
              [tune] - SORTINO 4w  (abs/rel): 1.48 / 1.74
              [tune] - SORTINO 13w (abs/rel): 0.79 / 0.89
              [tune] - MAR (CAGR/MDD): 1.07
              [tune] - MDD: 36.43%
              [tune] - CAGR: 38.86%
```

Then, the winning configs are backtested on the holdout period of days as a
"go live" simulation. The holdout period of days starts on the next candle
after the end of the tuner_days period.

```text
              [holdout] GrossVhodl: best
key metric! > [holdout] - LINKUSDT: +56.74%  TRADES: 44
              [holdout] - EDGE%: +7.91%, HODL%: -31.15%
              [holdout] - BUYS: 25, SELLS: 19
              [holdout] RiskStats
              [holdout] - SHARPE 1w  (abs/rel): 0.31 / 1.32
              [holdout] - SHARPE 4w  (abs/rel): 0.47 / 1.65
              [holdout] - SHARPE 13w (abs/rel): 1.61 / 0.19
              [holdout] - SORTINO 1w  (abs/rel): 0.47 / 1.87
              [holdout] - SORTINO 4w  (abs/rel): 0.68 / 2.39
              [holdout] - SORTINO 13w (abs/rel): 2.29 / 0.27
              [holdout] - MAR (CAGR/MDD): 0.27
              [holdout] - MDD: 35.56%
              [holdout] - CAGR: 9.49%
```

The key metric indicated above is the % increase in gross value vs benchmark.
The benchmark is a simple buy and hold investment for the same period and asset.

In this case, the holdout performance is relatively similar to the tuner
performance, indicating a good dynamic adaptability of the strategy to new
market conditions.

## Logbook
### Notes
One of the original ideas was to flash the algorithm to an MCU. However, after
consideration of resources, the use of a microcontroller restricts the
trading algorithm and the effort of developing and maintaining
a C codebase for the MCU would be better spent on refining the strategy and
deploying it on a cloud environment. I've yet to clean the codebase of the
guardrails pertaining to the MCU flashing.

The other .md files of the repo are used for LLM training. The codebase
has grown to a point where it's not feasible to have the agent scan the entire
repo for context. So the files in /docs are used to provide a high level
overview of the project and its structure. They are also used to document
design decisions. Keep in mind, the agent wrote them.

The oracle engine is a work in progress. Its objective is to identify some
wide configuration parameters that are likely to perform well on a new asset.
It does this non-causally using a slightly different set of logic. The idea is
then to use the non-causal parameters in a causal backtest environment to
narrow in on a best config of a new asset.

### Codex Workflow
On average, with a codebase this large and Codex on high mode, it takes around
5 to 15 minutes to perform simple edits with a good prompt and low familiarity
of codebase. This may be dependent on internet connection, but as the codebase
has grown, I've had to wait longer. Prompting the agent to become familiar with
the repo on high mode nukes the context to 50%.

Even just now, preparing the repo for GitHub, the agent fell into a loop of
testing some recent changes, and updating the README files, which took over an
hour. This was the result of a very simple prompt: "prepare all .md files by
aligning them to the current state of the codebase". The changes it was
accounting for were minor.

The most annoying behaviour I have experienced is the agent's tendency to
introduce excessive fallback and error handling code. While good in theory, it
has led to very messy and long debugging sessions. For example, when adding a
new logic gate or tuning key, the agent will include fallback values and variables
to older keys and gates that are known as a working state of the codebase. So,
if the new key stalls the algorithm, outputs will still be generated on old
gates and keys, providing a very confusing visual analysis of data. I put this 
down to the agent's prioritisation of working code as opposed to letting errors 
arise when things are not aligned.

Additionally, the excessive wrapping of code in handlers instead of rewriting
the core logic. You basically end up with deeply nested wrappers of your
original code if you are not careful.

The flip side of this is that the agent is extremely efficient at making
architectural changes. For example, adding a new logic gate touches almost
every file of the codebase. It will do this quickly with ease for all Python code. 
Shell scripts and JSON configs however tend to get forgotten about and left out 
of the changes.

Everything from the agent is ugly, and everything works. I've learnt a lot more
than what is outlined here about agentic coding. Maybe one day I'll write a
more detailed account. For now, I hope this provides some insight into the
project and process.
